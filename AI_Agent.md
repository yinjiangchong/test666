## 第一部分：Agent 核心原理

---

### Q1. 什么是 AI Agent？和普通 LLM 调用有什么本质区别？

**A：**

**普通 LLM 调用**是一次性的"输入 → 输出"，是无状态的单轮推理，LLM 本身不会主动决定下一步做什么。

**AI Agent** 的核心是赋予 LLM **自主决策与行动的能力**，它具备以下四个关键要素：

| 要素 | 说明 |
|---|---|
| **感知（Perception）** | 接收环境输入：用户消息、工具返回值、外部数据 |
| **规划（Planning）** | 由 LLM 决定下一步做什么、调用哪个工具 |
| **行动（Action）** | 执行工具调用、写文件、查数据库、访问 API 等 |
| **记忆（Memory）** | 保存历史对话、中间结果、长期知识 |

**运行循环（Agent Loop）：**

```
用户输入
   │
   ▼
LLM 推理 → 决定行动（工具调用 / 直接回答）
   │
   ├─ 调用工具 → 获取结果 → 再次输入 LLM → 继续推理
   │
   └─ 满足终止条件 → 输出最终结果
```

**后端工程师视角类比：**

| 概念 | 后端类比 |
|---|---|
| Agent Loop | 事件驱动的消息处理循环 |
| Tool Call | 调用第三方 API / 内部微服务 |
| Memory | Session 状态 + 数据库持久化 |
| Planning | 业务流程引擎 / 状态机 |

### Q2. ReAct 框架是什么？Agent 如何通过它完成推理与行动？

**A：**

ReAct（**Re**asoning + **Act**ing）是目前最主流的 Agent 推理模式，核心思想是让 LLM 在每一步交替进行：

1. **Thought（思考）** — LLM 分析当前情况，规划下一步
2. **Action（行动）** — 决定调用哪个工具，传入什么参数
3. **Observation（观察）** — 获取工具返回结果
4. 重复以上循环，直到得出最终答案

**一次完整的 ReAct 执行过程示例（查询订单状态）：**

```
用户: 帮我查一下订单 ORD-2024-001 的当前状态

Thought: 我需要查询订单状态，应该调用订单查询工具
Action: query_order(order_id="ORD-2024-001")
Observation: {"status": "shipped", "tracking_no": "SF1234567890"}

Thought: 已获取订单状态，用户可能还想知道物流信息
Action: query_logistics(tracking_no="SF1234567890")
Observation: {"location": "上海转运中心", "estimated": "明天送达"}

Thought: 信息已足够，可以给出完整回答
Final Answer: 您的订单已发货，快递单号 SF1234567890，
             当前在上海转运中心，预计明天送达。
```

**关键工程要点：**
- **Prompt 设计至关重要**：需明确告诉 LLM 可用工具列表、工具描述、输出格式规范
- **Tool Schema 要清晰**：工具名称、参数说明、返回值格式都要在 Prompt 中定义好
- **最大步骤限制（max_iterations）**：防止 LLM 陷入无限循环，通常设置 5~15 步
- **Observation 截断**：工具返回内容过长时需要截断/摘要，防止超出 Context Window

**与后端经验的对比：**

ReAct 的 Thought-Action-Observation 循环，本质上类似于后端的**责任链模式**，每一步处理结果作为下一步的输入，直到满足终止条件。

### Q3. Agent 的 Planning 规划能力如何实现？有哪些主流模式？

**A：**

Planning 是 Agent 处理**复杂、多步骤任务**的核心能力。主流规划模式：

**1. ReAct（边思考边行动）**
- 适合：任务路径不明确，需要根据中间结果动态调整
- 优点：灵活、适应性强
- 缺点：步骤多时容易"跑偏"，难以并行

**2. Plan-and-Execute（先规划后执行）**
- 先由 LLM 生成完整任务计划（步骤列表），再逐步执行
- 适合：任务结构清晰、步骤可预见的场景
- 优点：步骤可审查、可并行执行部分子任务
- 缺点：计划生成后灵活性较差，中途难以调整
```
用户任务
   │
   ▼
[Planner LLM] 生成执行计划
   ├─ Step1: 搜索竞品信息
   ├─ Step2: 分析价格区间
   └─ Step3: 生成报告
   │
   ▼
[Executor Agent] 逐步执行计划
   │
   ▼
汇总输出
```

**3. Tree of Thoughts（思维树）**
- 对同一问题生成多条推理路径，评估后选择最优路径
- 适合：需要探索多种解法的复杂问题（代码调试、策略制定）
- 缺点：LLM 调用次数多，成本高

**4. Self-Refine（自我反思优化）**
- 初次生成结果后，Agent 自我评估，发现问题后再次优化
- 适合：写作、代码生成、报告生成等对质量要求高的场景

**工程化选型建议：**
- 简单线性任务 → ReAct
- 结构化多步任务 → Plan-and-Execute
- 质量敏感场景 → Self-Refine
- 探索性问题 → Tree of Thoughts（注意成本控制）

### Q4. Agent 的 Memory 记忆体系如何设计？

**A：**

Agent 的记忆体系分为**四个层次**，对应不同的时效性和存储方式：

| 记忆类型 | 时效 | 存储位置 | 对应后端概念 |
|---|---|---|---|
| **In-Context Memory** | 当前会话 | LLM Context Window | 请求级缓存 / 局部变量 |
| **External Short-term** | 会话级 | Redis | Session 存储 |
| **External Long-term** | 持久化 | 向量数据库 / 关系型 DB | 用户画像 / 知识库 |
| **Episodic Memory** | 历史经验 | 向量数据库 + 摘要 | 日志归档 + 全文检索 |

**核心工程挑战：Context Window 管理**

LLM 的 Context Window 是有限资源（如 GPT-4 128K tokens），长对话必须做**记忆压缩**：

```
策略一：滑动窗口
  只保留最近 N 轮对话，超出部分丢弃

策略二：摘要压缩
  超出阈值时，用 LLM 对历史对话生成摘要，
  摘要 + 近期对话共同作为上下文

策略三：重要性筛选
  对每条记忆打重要性分数，低分记忆优先淘汰（类似 LRU + 权重）
```

**长期记忆的写入与检索：**
- **写入时机**：对话结束后异步提取关键信息（用户偏好、重要结论），存入向量数据库
- **检索时机**：每次对话开始时，用当前问题检索相关历史记忆，注入 Prompt
- **去重与更新**：相似记忆需要合并，避免 Context 被重复信息占满

**后端工程师视角：**
Memory 体系的设计本质上就是**多级缓存架构**：L1（Context） → L2（Redis） → L3（向量DB/关系DB），与后端缓存分级思路完全一致。

### Q5. RAG 的整体架构是什么？与纯 LLM 问答的区别？

**A：**

**RAG（Retrieval-Augmented Generation，检索增强生成）** 解决的核心问题：
- LLM 知识有截止日期，无法获取最新信息
- LLM 无法访问私有/企业内部数据
- LLM 幻觉（Hallucination）问题

**RAG 完整架构：**

```
【离线索引阶段（Indexing Pipeline）】
  原始文档（PDF/Word/网页/数据库）
       │
       ▼
  文档解析 → 文本清洗 → 分块（Chunking）
       │
       ▼
  Embedding 模型 → 向量化
       │
       ▼
  写入向量数据库（Milvus / Chroma / Qdrant / Weaviate）

【在线检索阶段（Query Pipeline）】
  用户问题
       │
       ▼
  Query 向量化
       │
       ▼
  向量相似度检索 → Top-K 相关文档块
       │
       ▼
  构建 Prompt（问题 + 检索结果）
       │
       ▼
  LLM 生成回答
```

**关键参数选择：**

| 参数 | 建议 | 说明 |
|---|---|---|
| Chunk Size | 256~512 tokens | 太大语义噪音多，太小上下文不足 |
| Chunk Overlap | 50~100 tokens | 防止关键信息被切断 |
| Top-K | 3~5 | 检索文档数，过多会稀释有效信息 |
| Embedding 模型 | text-embedding-3-small / BGE | 中文优先 BGE 系列 |

### Q6. RAG 效果差怎么排查和优化？

**A：**

RAG 效果差可以分三个层次排查：**检索问题、生成问题、数据问题**。

**排查框架：**

```
RAG 效果差
   │
   ├─ 检索阶段问题（召回不准）
   │    ├─ 语义相似但内容无关 → Rerank 重排序
   │    ├─ 关键词匹配缺失    → 引入 BM25 混合检索
   │    ├─ 问题太短/模糊     → Query 改写/扩展
   │    └─ 向量维度不匹配   → 更换 Embedding 模型
   │
   ├─ 生成阶段问题（有料没用好）
   │    ├─ LLM 忽略检索结果  → Prompt 优化，强调引用来源
   │    ├─ 上下文太长        → 摘要压缩，精选最相关块
   │    └─ 幻觉严重          → 引入引用溯源，强制 LLM 标注来源
   │
   └─ 数据质量问题（根本原因）
        ├─ 文档解析有误（表格/图片丢失）→ 优化解析器
        ├─ 分块策略不合理 → 按语义/段落分块
        └─ 知识库过时      → 建立增量更新机制
```

**主要优化手段：**

| 优化点 | 手段 | 效果 |
|---|---|---|
| **Query 改写** | 用 LLM 将用户问题扩写成多个检索 Query | 提升召回率 |
| **混合检索** | 向量检索（语义） + BM25（关键词）融合 | 平衡精准率与召回率 |
| **Rerank 重排序** | 用 Cross-Encoder 对 Top-K 结果重新打分 | 显著提升精准率 |
| **语义分块** | 按段落/章节而非固定长度切分 | 避免语义被截断 |
| **HyDE** | 先用 LLM 生成假设答案，再用答案检索 | 提升语义匹配度 |

### Q7. 向量检索之外还有哪些检索增强手段？

**A：**

单纯的向量检索在很多场景下是不够的，生产级 RAG 通常采用**多路检索融合**：

**1. BM25 关键词检索**
- 对精确名词（人名、产品型号、专有名词）效果远好于向量检索
- 与向量检索结果通过 RRF（倒数排名融合）合并

**2. 知识图谱检索（GraphRAG）**
- 将文档中的实体和关系构建为知识图谱
- 适合需要多跳推理的问题（A→B→C 的关联关系）
- 微软开源的 GraphRAG 是代表性方案

**3. 结构化数据检索（Text2SQL）**
- 将自然语言问题转换为 SQL 查询，直接查关系型数据库
- 适合数据分析类问答（"上个月销售额最高的品类是？"）
**4. 元数据过滤（Metadata Filtering）**
- 在向量检索前，先用结构化字段过滤（时间范围、部门、文档类型）
- 大幅缩小检索范围，提升精准率和速度

**5. 父子文档检索（Parent-Child Retrieval）**
- 索引时用小块（Child）索引，检索命中后返回大块（Parent）上下文
- 解决"检索精准但上下文不足"的矛盾

**实际工程推荐组合：**
```
向量检索（语义）+ BM25（关键词）+ 元数据过滤
              │
              ▼
         RRF 融合排序
              │
              ▼
        Rerank 精排（Cross-Encoder）
              │
              ▼
         Top-3 送入 LLM
```

### Q8. LangChain 的核心抽象是什么？什么场景适合用它？

**A：**

LangChain 的核心是**链式组合（Chain）** 和**统一抽象层**的思想：

**六大核心抽象：**

| 抽象 | 作用 |
|---|---|
| **LLM / ChatModel** | 统一封装各家 LLM（OpenAI、Anthropic、本地模型） |
| **Prompt Template** | 结构化 Prompt 管理，支持变量注入 |
| **Chain** | 将多个组件串联成处理管道（类似 Pipeline） |
| **Tool / ToolKit** | 封装外部能力（搜索、代码执行、数据库查询） |
| **Memory** | 会话记忆管理 |
| **Agent** | 基于 ReAct/OpenAI Function Call 的决策执行器 |

**LangChain Expression Language（LCEL）** 是新一代核心：
- 用 `|` 管道符连接组件，天然支持流式输出、并行执行
- 类似 Unix 管道，符合 Python 开发者直觉

**适合使用 LangChain 的场景：**
- 快速原型验证，不想从零搭建 RAG / Agent 基础设施
- 需要频繁切换不同 LLM 提供商（统一接口的价值）
- 生态丰富的工具集成（100+ 预置工具）
- 中小型 Agent 应用，复杂度可控

**不适合的场景：**
- 超高并发生产系统（LangChain 抽象层性能开销不可忽视）
- 需要极度定制化的 Agent 行为
- 团队对框架黑盒行为难以 debug 和掌控

### Q9. AutoGen 和 CrewAI 的设计理念有什么差异？如何选型？

**A：**

| 维度 | AutoGen（微软） | CrewAI |
|---|---|---|
| **核心理念** | 多 Agent 对话协作，Agent 之间互相对话驱动任务 | 角色扮演，每个 Agent 有明确职责和角色定位 |
| **编排模式** | 自由对话式（Agent 之间自主协商） | 流程编排式（Manager 分配任务给 Worker） |
| **灵活性** | 高，适合探索性任务 | 中，结构更清晰 |
| **可控性** | 较低，对话流向难以精确控制 | 较高，任务流程可预定义 |
| **适用场景** | 代码生成与审查、研究类任务、需要 Agent 互辩 | 内容创作流水线、有明确角色分工的业务流程 |
| **学习成本** | 中 | 低 |

**选型建议：**

```
任务类型
   │
   ├─ 探索性 / 对话驱动 / 代码生成调试 → AutoGen
   │
   ├─ 流程固定 / 角色明确 / 内容生产   → CrewAI
   │
   └─ 超高定制化 / 生产级性能要求       → 自研 Agent 框架
```

**后端工程师视角：**
- AutoGen ≈ **Actor 模型**（每个 Agent 是独立 Actor，互相发消息）
- CrewAI ≈ **Worker 线程池**（Manager 分发任务，Worker 执行后汇报）

### Q10. 框架之外，什么时候应该"裸写" Agent？

**A：**

框架（LangChain/AutoGen/CrewAI）提升了开发速度，但在以下场景应考虑**绕开框架或自研**：

| 场景 | 原因 |
|---|---|
| **超高并发生产系统** | 框架封装层引入性能开销，难以精细控制并发和连接池 |
| **复杂定制化流程** | 框架的抽象反而成为约束，需要大量"逆向使用"框架 |
| **极致成本控制** | 需要精确控制每次 LLM 调用的 Token 用量 |
| **安全敏感场景** | 框架的自动 Prompt 拼接可能引入注入风险，需要精确审计 |
| **已有成熟后端基础设施** | 你的 FastAPI + Redis + Kafka 体系已经很完善，框架带来的是束缚而非加速 |

**裸写 Agent 的核心组件清单：**
- LLM Client 封装（OpenAI SDK / 本地模型）
- Tool 注册与调度机制
- Prompt 模板管理
- ReAct Loop 实现
- Memory 管理
- 错误处理与重试

**10年后端的优势：** 裸写 Agent 其实就是在做**一套有 LLM 决策节点的业务系统**，你的微服务、消息队列、数据库、缓存经验在这里全部可以复用。

## 第四部分：多 Agent 系统设计与编排

---

### Q11. 多 Agent 系统的核心设计模式有哪些？

**A：**

多 Agent 系统解决的问题是：**单个 Agent 能力边界有限**，通过协作完成更复杂的任务。

**四种核心编排模式：**

**1. Sequential（顺序流水线）**
```
Agent A → Agent B → Agent C → 最终输出
```
- 每个 Agent 处理上一个的输出
- 适合：有明确先后依赖的任务（研究→写作→校对）

**2. Parallel（并行执行）**
```
          ┌─ Agent A（子任务1）─┐
主任务 ──►│─ Agent B（子任务2）─│──► 汇总 Agent → 最终输出
          └─ Agent C（子任务3）─┘
```
- 多个 Agent 并行处理独立子任务
- 适合：子任务互相独立，对延迟敏感的场景
- 后端类比：**并发 Future / asyncio.gather**

**3. Supervisor（主管模式）**
```
Supervisor Agent（决策）
   │
   ├─ 分配给 Agent A
   ├─ 分配给 Agent B
   └─ 汇总所有结果 → 输出
```
- Supervisor 动态决定调用哪个子 Agent
- 适合：任务类型多样，需要动态路由

**4. Hierarchical（层级式）**
```
顶层 Orchestrator
   ├─ Mid-level Manager A
   │    ├─ Worker Agent 1
   │    └─ Worker Agent 2
   └─ Mid-level Manager B
        ├─ Worker Agent 3
        └─ Worker Agent 4
```
- 适合：超大规模复杂任务，需要多层分治

---

### Q12. 如何做多 Agent 之间的任务拆解与分配？

**A：**

任务拆解是多 Agent 系统中 **最关键也是最容易出问题**的环节。

**任务拆解的核心原则：**

- **子任务原子性**：每个子任务应该职责单一，输入输出清晰
- **依赖关系显式化**：哪些任务可以并行，哪些有先后依赖，需要明确定义
- **边界清晰**：子 Agent 不应感知全局任务，只处理自己分配的部分

**两种拆解方式：**

| 方式 | 说明 | 优缺点 |
|---|---|---|
| **静态拆解** | 预定义任务分解规则，由代码控制流程 | 可控、可预测，但灵活性低 |
| **动态拆解** | 由 Planner LLM 根据任务内容动态生成子任务列表 | 灵活，但稳定性依赖 LLM 规划质量 |

**任务分配策略：**

```
Planner 生成任务列表
       │
       ▼
任务队列（Redis / 内存队列）
       │
       ├─ 根据 Agent 能力标签匹配
       ├─ 负载均衡（轮询 / 最少任务）
       └─ 优先级调度（关键路径任务优先）
```

**后端工程师天然优势：**
这套机制就是一个**带 LLM 决策节点的任务调度系统**，与 Celery 任务队列 + Worker 的架构几乎同构，你的任务队列经验可以直接复用。

---

### Q13. 多 Agent 协作中如何保证结果的一致性与可靠性？

**A：**

多 Agent 协作的可靠性是生产落地的最大挑战，主要面临以下问题：

**核心挑战与解决方案：**

**1. 中间结果不可靠（LLM 输出格式不稳定）**
- 方案：对每个 Agent 的输出使用 **结构化输出（Structured Output）**，强制 JSON Schema 校验
- 方案：每个 Agent 节点加入**输出验证层**，不合格则重试

**2. 长链路中错误传播**
- 方案：关键节点设置**检查点（Checkpoint）**，将中间状态持久化（Redis/DB）
- 方案：支持从失败节点**断点续传**，而非整体重跑
- 后端类比：Saga 模式的补偿机制

**3. Agent 之间上下文传递混乱**
- 方案：定义统一的**任务上下文对象（Task Context）**，所有 Agent 读写同一个结构化状态
- 方案：使用**共享状态存储（Redis）** 而非 Agent 间直接传参

**4. 循环调用与死锁**
- 方案：设置**最大调用深度**限制
- 方案：用有向无环图（DAG）定义 Agent 调用拓扑，从设计上规避循环
**整体可靠性架构：**
```
任务入口
   │
   ▼
持久化任务状态（Redis / DB）
   │
   ▼
Agent 执行 → 输出校验 → 检查点保存
   │                         │
   │ 失败                     ▼
   └──────────── 重试 / 降级处理
   │
   ▼
下一 Agent（读取检查点继续）
```

### Q14. Agent 流式输出如何实现？后端如何处理？

**A：**

用户与 Agent 交互时，等待 LLM 完整生成后再返回体验极差，**流式输出**是生产必备。

**流式输出架构：**

```
LLM API（SSE / WebSocket）
       │  token by token 流式返回
       ▼
Agent 处理层
  ├─ 普通文本：直接透传给客户端
  ├─ Tool Call：等待完整 JSON 再执行工具
  └─ 工具执行中：推送"思考中..."等状态提示
       │
       ▼
FastAPI StreamingResponse / WebSocket
       │
       ▼
前端实时渲染
```

**关键工程细节：**

| 问题 | 解决方案 |
|---|---|
| Tool Call 时无法流式 | 工具执行期间推送中间状态（"正在查询数据库..."） |
| 多步 Agent 中间结果 | 每步完成后推送阶段性结果，而非等全部结束 |
| 网络中断重连 | SSE 天然支持断点续传（Last-Event-ID） |
| 长时间无输出心跳 | 定期发送心跳事件，防止连接超时 |

**协议选型：**
- **SSE（Server-Sent Events）**：单向推送，实现简单，适合大多数 Agent 场景
- **WebSocket**：双向通信，适合需要用户中途打断 Agent 的场景

**后端工程师注意：** 流式输出下，Nginx/网关的**超时配置**需要调整（`proxy_read_timeout`），否则长时间推理会被中间件断连。

### Q15. 如何控制 Token 消耗与 LLM 调用成本？

**A：**

Token 成本是 Agent 系统生产落地的核心约束，10年后端的**成本意识**在这里非常重要。

**Token 消耗的主要来源：**

```
总 Token 消耗
   ├─ System Prompt（每次请求都会发送）
   ├─ 历史对话（随会话增长）
   ├─ 检索文档（RAG 注入）
   ├─ Tool Schema 描述（每次请求）
   └─ 模型输出（Output Token 通常比 Input 贵 3x）
```

**成本优化策略：**

| 策略 | 手段 | 效果 |
|---|---|---|
| **Prompt 压缩** | 用小模型压缩 System Prompt 和历史对话 | 减少 30~50% Input Token |
| **模型分级路由** | 简单任务用小模型（GPT-3.5/Haiku），复杂任务才用大模型 | 降低 60~80% 成本 |
| **缓存 LLM 结果** | 相同/相似 Prompt 命中缓存直接返回（Semantic Cache） | 对重复查询效果显著 |
| **Tool Schema 优化** | 只注入当前任务相关的工具，而非全量工具列表 | 减少 10~30% Token |
| **批处理合并** | 多个短任务合并为一次 LLM 调用 | 减少调用次数 |
| **Output 限制** | 明确在 Prompt 中限制输出长度和格式 | 控制 Output Token |

**模型分级路由示例：**
```
任务复杂度评估
   │
   ├─ 简单问答 / 格式提取   → GPT-3.5-turbo / Claude Haiku
   ├─ 中等推理 / 代码生成   → GPT-4o-mini / Claude Sonnet
   └─ 复杂规划 / 多步推理   → GPT-4o / Claude Opus
```

**成本监控必备指标：**
- 每请求平均 Token 消耗
- 每用户日均 Token 消耗
- 模型调用次数与成功率
- 缓存命中率

### Q16. Agent 系统的可观测性如何建设？

**A：**

Agent 系统的可观测性比普通后端更复杂，因为**LLM 的推理过程是黑盒**，需要专门的手段来观测。

**三大支柱 + Agent 特有维度：**

| 支柱 | 普通后端 | Agent 系统额外关注 |
|---|---|---|
| **Metrics** | QPS、延迟、错误率 | Token 用量、LLM 调用次数、工具调用成功率、每步耗时 |
| **Tracing** | 请求链路追踪 | Agent 完整推理链路（Thought→Action→Observation 每步记录） |
| **Logging** | 结构化日志 | Prompt 输入/输出完整记录、工具调用参数与返回值 |

**Agent 专用可观测性工具：**

| 工具 | 用途 |
|---|---|
| **LangSmith** | LangChain 官方追踪平台，可视化 Agent 执行每一步 |
| **LangFuse** | 开源 LLM 可观测性平台，支持 Prompt 版本管理 |
| **Arize Phoenix** | Agent 链路追踪 + 评估，支持多框架 |
| **OpenTelemetry** | 标准化 Trace 方案，与现有后端监控体系打通 |

**关键告警规则：**
- Agent 平均步骤数异常升高（可能陷入循环）
- 单次请求 Token 超阈值（防止异常消耗）
- 工具调用失败率 > 5%
- LLM 返回非结构化输出（解析失败率）
- 端到端延迟 P99 超阈值

### Q17. Agent 调用链如何做容错、重试与降级？

**A：**

Agent 系统面临的错误类型比普通后端更多样，需要**分层容错**：

**错误分类与处理策略：**

| 错误类型 | 示例 | 处理策略 |
|---|---|---|
| **LLM API 错误** | 限流（429）、超时、服务不可用 | 指数退避重试 + 备用模型 Fallback |
| **输出格式错误** | LLM 返回非 JSON，解析失败 | 重试 + 修正 Prompt + 最多 3 次 |
| **工具调用错误** | 工具超时、返回异常 | 重试 + 错误信息回传 LLM，让其调整策略 |
| **无限循环** | Agent 重复调用同一工具 | 最大步骤数限制 + 循环检测 |
| **上下文溢出** | Context Window 超限 | 自动触发记忆压缩 |

**重试策略设计：**

```
LLM 调用失败
   │
   ▼
第1次重试（立即，同模型）
   │ 失败
   ▼
第2次重试（延迟 2s，同模型）
   │ 失败
   ▼
第3次重试（延迟 4s，降级到备用模型）
   │ 失败
   ▼
返回降级响应（友好错误信息）+ 异步告警
```

**降级策略：**
- **模型降级**：主模型（GPT-4o）→ 备用模型（GPT-3.5）→ 本地模型
- **功能降级**：Agent 模式失败 → 退化为普通 RAG 问答 → 退化为关键词搜索
- **响应降级**：无法完成任务时，返回部分结果 + 说明原因，而非直接报错

**后端工程师类比：**
与微服务中的**熔断器（Circuit Breaker）+ 服务降级**完全同构，Hystrix / Resilience4j 的设计思想直接适用。

### Q18. 如何设计一个生产可用的客服 Agent？

**A：**

客服 Agent 是最典型的落地场景，核心挑战是：**准确性要求高 + 边界要控制严 + 用户体验要好**。

**整体架构设计：**

```
用户输入
   │
   ▼
[意图识别层]
  ├─ 闲聊/情绪安抚    → 直接由 LLM 回复（无需 RAG）
  ├─ 产品咨询        → RAG 检索知识库
  ├─ 订单/账户操作   → 调用业务 API 工具
  └─ 投诉/超出能力   → 路由到人工坐席
   │
   ▼
[客服 Agent 执行层]
  ├─ 知识库 RAG（产品文档、FAQ、政策）
  ├─ 业务工具（查订单、退款申请、地址修改）
  └─ 对话记忆（本次会话上下文）
   │
   ▼
[安全过滤层]
  ├─ 敏感词过滤
  ├─ 竞品信息屏蔽
  └─ 越权操作拦截
   │
   ▼
输出给用户
```

**关键设计原则：**

| 原则 | 实践 |
|---|---|
| **能力边界清晰** | 明确 Agent 能做什么、不能做什么，超出边界主动转人工 |
| **操作需确认** | 涉及写操作（退款、修改）必须让用户二次确认 |
| **来源可追溯** | 回答引用来源，防止幻觉导致错误信息 |
| **无缝转人工** | Agent 解决不了时，将完整上下文传递给人工坐席 |
| **话术一致性** | 统一 System Prompt 规范语气、品牌调性 |
**质量评估体系：**
- 意图识别准确率
- 问题解决率（FCR，First Contact Resolution）
- 转人工率（过高说明 Agent 能力不足）
- 用户满意度（CSAT）
- 幻觉率（回答与知识库不符的比例）

### Q19. 如何设计代码生成 Agent？核心挑战在哪里？

**A：**

代码生成 Agent 的核心挑战在于：**代码要能真正运行，而不只是"看起来对"**。

**架构设计：**

```
用户需求描述
   │
   ▼
[需求理解 Agent]
  └─ 澄清需求、拆解子任务、确认技术约束
   │
   ▼
[代码生成 Agent]
  └─ 生成初版代码（含注释和类型注解）
   │
   ▼
[代码执行 Agent]（沙箱环境）
  └─ 实际运行代码，捕获报错信息
   │
   ├─ 执行成功 → 输出结果
   └─ 执行失败 → 错误信息回传
   │
   ▼
[代码修复 Agent]（自我修复循环）
  └─ 基于报错信息修复代码，最多 N 次迭代
   │
   ▼
[代码审查 Agent]
  └─ 安全检查、代码规范、边界条件
   │
   ▼
最终代码输出
```

**核心挑战与解决方案：**

| 挑战 | 解决方案 |
|---|---|
| **代码执行安全** | 沙箱隔离（Docker 容器/e2b/Dask），限制文件系统和网络访问 |
| **依赖管理** | 预置常用库的隔离环境，动态安装需审批 |
| **自修复循环** | 设置最大修复次数（3~5次），超出后返回当前最优版本 + 说明 |
| **上下文丢失** | 多文件项目需要维护代码库上下文（RAG 索引已有代码） |
| **测试验证** | 自动生成单元测试并运行，以测试通过率评估代码质量 |

**10年后端的优势：**
- 对代码质量、工程规范有深刻理解，能写出更好的 Prompt 约束代码风格
- 懂沙箱、容器、安全隔离，能设计更安全的执行环境
- 懂 CI/CD，能将代码生成 Agent 集成到开发工作流

### Q20. 如何设计数据分析 Agent？

**A：**

数据分析 Agent（Text-to-Analysis）让非技术用户能用自然语言分析数据，核心是 **自然语言 → 数据操作 → 可视化结果**。

**架构设计：**

```
自然语言问题
（"上个季度各地区的销售额对比"）
   │
   ▼
[意图解析 Agent]
  └─ 理解分析意图、所需指标、时间维度、分组维度
   │
   ▼
[Schema 感知层]
  └─ RAG 检索数据库表结构，理解字段含义和业务含义
   │
   ▼
[Text2SQL Agent]
  └─ 生成 SQL / Pandas 代码
   │
   ▼
[SQL 安全检查层]
  └─ 拦截 DDL（DROP/ALTER）、超大查询（LIMIT 兜底）
   │
   ▼
[SQL 执行层]
  └─ 只读连接执行，返回结构化数据
   │
   ├─ 执行失败 → SQL 修复循环
   └─ 执行成功 ↓
   │
   ▼
[分析解读 Agent]
  └─ 对结果数据进行自然语言解读，发现趋势和异常
   │
   ▼
[可视化 Agent]
  └─ 生成图表代码（ECharts / Matplotlib / Plotly）
   │
   ▼
自然语言分析报告 + 可视化图表
```
**关键工程点：**

| 要点 | 说明 |
|---|---|
| **Schema 管理** | 维护数据库字段的业务语义描述（字段注释），而非只靠字段名 |
| **SQL 安全** | 只读账户 + 查询行数上限 + DDL 拦截，防止误操作 |
| **SQL 修复** | 执行失败将报错信息回传 LLM 修复，最多 3 次 |
| **多步分析** | 复杂问题拆分为多个 SQL，中间结果可以被后续查询引用 |
| **结果可信度** | 对生成的 SQL 和分析结论展示给用户审查，而非完全黑盒 |

**与后端经验的结合：**
- 你对 SQL 优化（索引、JOIN 策略、分页）的理解，可以用于审查和修正 LLM 生成的 SQL
- 数据库连接池、慢查询监控等后端基础设施可以直接复用
- 对业务数据模型的理解，能帮你写出更精准的 Schema 描述，提升 Text2SQL 准确率

| 考点 | 核心要点 | 关键词 |

| **AutoGen vs CrewAI** | 对话协作 vs 角色编排 | Actor模型 vs 任务队列 |
| **多Agent模式** | Sequential / Parallel / Supervisor / Hierarchical | DAG、检查点 |
| **任务拆解** | 原子性、依赖显式化、静态vs动态 | 任务队列、Celery同构 |
| **流式输出** | SSE / WebSocket，工具执行期推送状态 | StreamingResponse |
| **成本控制** | Prompt压缩、模型分级路由、语义缓存 | Token 监控、Semantic Cache |
| **可观测性** | Metrics+Tracing+Logging，LangSmith/LangFuse | Agent 专用链路追踪 |
| **容错降级** | 分层重试、模型降级、功能降级 | Circuit Breaker、Saga |
| **客服 Agent** | 意图识别分流、边界控制、转人工 | FCR、幻觉率 |
| **代码生成 Agent** | 沙箱执行、自修复循环、测试验证 | e2b、安全隔离 |
| **数据分析 Agent** | Text2SQL、Schema 感知、只读安全、SQL 修复 | 业务语义、DDL 拦截 |
